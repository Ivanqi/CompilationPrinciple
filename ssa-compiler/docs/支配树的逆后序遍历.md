# 控制流图示例
```
支配树结构：
块0 (ENTRY)
├── 块1
│   ├── 块3
│   └── 块4
└── 块2
    └── 块5
```

# 遍历过程分析
## 步骤1：从块0开始
```rust
postorder(cfg, 0, &mut res)
```

## 步骤2：递归处理块0的被支配子块
```rust
// 先处理块1
postorder(cfg, 1, &mut res)
// 在块1中，先处理块3
postorder(cfg, 3, &mut res)  // 块3没有子块，直接push(3)
// 然后处理块4  
postorder(cfg, 4, &mut res)  // 块4没有子块，直接push(4)
// 最后push块1
push(1)
```

## 步骤3: 继续处理块0的其他子块
```rust
// 处理块2
postorder(cfg, 2, &mut res)
// 在块2中，先处理块5
postorder(cfg, 5, &mut res)  // 块5没有子块，直接push(5)
// 然后push块2
push(2)
```

## 步骤4: 最后push 块0
```rust
push(0)
```

## 最后遍历结果
```rust
res = [3, 4, 1, 5, 2, 0]
```

# 算法原理详解
## 1. 后序遍历(Post-Order)
```rust
// 后序遍历顺序：左子树 → 右子树 → 根节点
for &child in &block.idom_of {
    postorder(s, child, acc);  // 先递归处理所有子节点
}
acc.push(current);             // 再访问当前节点
```

## 2. 逆后序(Reverse Post-Order)
```
// 后序遍历结果：[3, 4, 1, 5, 2, 0]
// 逆后序结果：  [0, 2, 5, 1, 4, 3] (实际代码返回的就是后序，注释说明逆后序)
```

## 3. 支配树遍历
```
// block.idom_of 包含被当前块直接支配的所有块
// 这反映了支配树的父子关系
```

# 在编译器优化中的应用
## 1. GVN-PRE优化
```rust
// 在generate_value_table中使用
let mut rpo = cfg.get_dom_rpo();
while let Some(next) = rpo.pop() {
    // 按逆后序处理基本块，确保依赖关系正确
    let block = cfg.get_block(next);
    for op in &block.body {
        value_table.maybe_insert_op(op, ...);
    }
}
```

## 2. 数据流分析
```
// 逆后序遍历确保在处理表达式时，其依赖的值已经被编号
// 例如：块1依赖于块3和块4，在逆后序中块3和块4先被处理
```

## 3. 支配性质保证
```
// 逆后序遍历保证：
// - 支配者总是在被支配者之前被处理
// - 循环依赖得到正确处理
// - 数据流分析能够收敛
```
