# `build_interference_graph` 函数分析报告

## 函数作用概述

`build_interference_graph` 函数是**寄存器分配**中构建冲突图的核心实现。该函数的主要作用是分析控制流图中虚拟寄存器之间的冲突关系，构建冲突图（Interference Graph），同时计算可合并的寄存器对和溢出权重。

具体来说，函数的主要功能是：
1. **构建冲突图**：分析哪些虚拟寄存器同时活跃（冲突）
2. **识别可合并寄存器**：找出可以合并的寄存器对（如复制指令）
3. **计算溢出权重**：为每个寄存器计算溢出到栈上的代价权重

## 每段代码详细注释

```rust
fn build_interference_graph(
    cfg: &CFG<Operator>,
) -> (
    InterferenceGraph,
    HashSet<(VReg, VReg)>,
    HashMap<VReg, usize>,
) {
    // 步骤1：初始化数据结构
    // graph: 冲突图，存储寄存器之间的冲突关系
    let mut graph = InterferenceGraph::new();
    
    // coalescable: 可合并的寄存器对集合
    // 例如：(r1, r2) 表示 r1 和 r2 可以合并为同一个寄存器
    let mut coalescable = HashSet::new();
    
    // spill_weights: 溢出权重映射表
    // 键：寄存器编号，值：溢出到栈上的代价权重
    let mut spill_weights = HashMap::new();
    
    // 步骤2：计算每个基本块的活跃出集合（live-out）
    // live_out[i] 表示基本块i出口处活跃的寄存器集合
    let live_out = cfg.live_out();
    
    // 步骤3：遍历所有基本块，构建冲突图
    for (i, block) in cfg.get_blocks().iter().enumerate() {
        // 初始化当前活跃集合为基本块出口处的活跃集合
        let mut live_now = live_out[i].clone();
        
        // 步骤4：反向遍历基本块中的指令（从后往前）
        // 反向遍历是因为活跃变量分析是反向数据流分析
        for op in block.body.iter().rev() {
            // 步骤4.1：为当前活跃的每个寄存器增加溢出权重
            // 寄存器在更多指令中活跃，溢出代价越高
            for &live in &live_now {
                *spill_weights.entry(live).or_insert(0_usize) += 1;
            }
            
            // 步骤4.2：处理指令的结果寄存器（如果有的话）
            if let Some(rec) = op.receiver() {
                // 将结果寄存器添加到冲突图中
                graph.maybe_insert(rec);
                
                // 结果寄存器被定义，从当前活跃集合中移除
                // 因为定义会"杀死"之前的活跃值
                live_now.remove(&rec);
                
                // 步骤4.3：特殊处理复制指令（Mv）
                if let Operator::Mv(lr1, lr2) = op {
                    // 复制指令表示两个寄存器可以合并
                    // 例如：Mv(r1, r2) 表示 r1 是 r2 的复制
                    coalescable.insert((*lr1, *lr2));
                    
                    // 对于复制指令，结果寄存器与所有活跃寄存器冲突
                    // 除了源寄存器（因为它们是同一个值）
                    live_now
                        .iter()
                        .filter(|&lr| lr != lr2)  // 排除源寄存器
                        .for_each(|&lr| graph.add_edge(lr, rec));
                } else {
                    // 对于非复制指令，结果寄存器与所有活跃寄存器冲突
                    live_now.iter().for_each(|&lr| graph.add_edge(lr, rec));
                }
            }
            
            // 步骤4.4：将指令的依赖寄存器添加到当前活跃集合
            // 这些寄存器在指令执行前必须是活跃的
            live_now.extend(op.dependencies());
        }
    }
    
    // 步骤5：返回构建的结果
    (graph, coalescable, spill_weights)
}
```

## 算法流程详解

### 步骤1：初始化数据结构
- `graph`：冲突图，用于存储寄存器之间的冲突关系
- `coalescable`：可合并寄存器对集合
- `spill_weights`：溢出权重映射表

### 步骤2：计算活跃出集合
- 调用 `cfg.live_out()` 计算每个基本块出口处的活跃寄存器集合
- 这是反向数据流分析的结果

### 步骤3：遍历基本块构建冲突图
1. 对每个基本块，从出口处的活跃集合开始
2. 反向遍历基本块中的指令
3. 为每个指令更新冲突关系和活跃集合

### 步骤4：反向遍历指令
1. **更新溢出权重**：为当前活跃的每个寄存器增加权重
2. **处理结果寄存器**：
   - 将结果寄存器添加到冲突图
   - 从活跃集合中移除结果寄存器（被定义杀死）
   - 添加冲突边：结果寄存器与所有当前活跃寄存器冲突
3. **特殊处理复制指令**：
   - 标记可合并的寄存器对
   - 结果寄存器与所有活跃寄存器冲突（除了源寄存器）
4. **更新活跃集合**：将指令的依赖寄存器添加到活跃集合

## 举例说明

### 示例代码
```
基本块B0:
  r1 = 10          // li r1, 10
  r2 = r1 + 5      // add r2, r1, 5
  r3 = r2          // mv r3, r2
  r4 = r3 * 2      // mult r4, r3, 2
```

### 步骤1：计算活跃出集合
假设基本块B0的活跃出集合为空（没有后继基本块）

### 步骤2：反向遍历构建冲突图

#### 指令4：`mult r4, r3, 2`
- 当前活跃集合：`{}`
- 结果寄存器：`r4`
- 依赖寄存器：`r3, 2`（立即数2忽略）
- 操作：
  - 添加 `r4` 到冲突图
  - 活跃集合添加 `r3` → `{r3}`

#### 指令3：`mv r3, r2`
- 当前活跃集合：`{r3}`
- 结果寄存器：`r3`
- 依赖寄存器：`r2`
- 操作：
  - 为活跃寄存器增加权重：`spill_weights[r3] += 1`
  - 添加 `r3` 到冲突图
  - 从活跃集合移除 `r3` → `{}`
  - 标记可合并：`coalescable.insert((r3, r2))`
  - 活跃集合添加 `r2` → `{r2}`

#### 指令2：`add r2, r1, 5`
- 当前活跃集合：`{r2}`
- 结果寄存器：`r2`
- 依赖寄存器：`r1, 5`
- 操作：
  - 为活跃寄存器增加权重：`spill_weights[r2] += 1`
  - 添加 `r2` 到冲突图
  - 从活跃集合移除 `r2` → `{}`
  - 添加冲突边：`r2` 与所有活跃寄存器冲突（当前无）
  - 活跃集合添加 `r1` → `{r1}`

#### 指令1：`li r1, 10`
- 当前活跃集合：`{r1}`
- 结果寄存器：`r1`
- 依赖寄存器：无
- 操作：
  - 为活跃寄存器增加权重：`spill_weights[r1] += 1`
  - 添加 `r1` 到冲突图
  - 从活跃集合移除 `r1` → `{}`

### 步骤3：最终结果
- **冲突图**：
  - `r1` 与 `r2` 冲突（在指令2处同时活跃）
  - `r2` 与 `r3` 冲突（在指令3处同时活跃）
  - `r3` 与 `r4` 冲突（在指令4处同时活跃）
- **可合并集合**：`{(r3, r2)}`
- **溢出权重**：
  - `r1`: 1
  - `r2`: 1
  - `r3`: 1
  - `r4`: 0

## 在寄存器分配中的位置

`build_interference_graph` 函数在寄存器分配流程中的位置：

1. **SSA形式转换**：`conventionalize_ssa()`
2. **活跃范围重写**：`rewrite_liveranges()`
3. **构建冲突图**：`build_interference_graph()`（当前函数）
4. **图着色分配**：`find_coloring()`
5. **溢出处理**：`spill_liverange()`
6. **寄存器替换**：将虚拟寄存器替换为物理寄存器

## 性能特点

- **时间复杂度**：O(n × m)，其中n是基本块数量，m是指令数量
- **空间复杂度**：O(v²) 最坏情况，其中v是寄存器数量
- **关键优化**：反向遍历指令，高效更新活跃集合

## 算法设计的核心思想

### 1. 冲突图构建原理
- 两个寄存器在同一个程序点都活跃，则它们冲突
- 冲突的寄存器不能分配到同一个物理寄存器

### 2. 可合并寄存器识别
- 复制指令 `Mv(r1, r2)` 表示 `r1` 和 `r2` 可以合并
- 合并可以减少寄存器使用，提高分配成功率

### 3. 溢出权重计算
- 寄存器在更多指令中活跃，溢出代价越高
- 用于智能选择要溢出的寄存器

### 4. 反向遍历策略
- 活跃变量分析是反向数据流问题
- 从出口向入口遍历，自然实现活跃变量分析

## 关键技术细节

### 冲突边添加规则
1. **普通指令**：结果寄存器与所有当前活跃寄存器冲突
2. **复制指令**：结果寄存器与所有当前活跃寄存器冲突（除了源寄存器）
3. **依赖寄存器**：在指令执行前必须活跃

### 活跃集合更新
1. **定义杀死**：结果寄存器定义时，从活跃集合移除
2. **使用生成**：依赖寄存器使用时，添加到活跃集合
3. **反向传播**：从出口向入口传播活跃信息

### 溢出权重意义
- 权重越高，寄存器在更多指令中活跃
- 溢出高权重寄存器代价更大
- 用于选择溢出代价最小的寄存器

## 举例：循环中的冲突

```
while (i > 0) {
  r1 = r1 + 1
  r2 = r2 * 2
}
```

在这个循环中：
- `r1` 和 `r2` 在整个循环中都活跃
- 它们之间添加冲突边
- 溢出权重较高（在循环中多次使用）

## 与图着色算法的关系

构建的冲突图直接用于图着色算法：
1. **节点**：虚拟寄存器
2. **边**：寄存器之间的冲突关系
3. **着色**：为每个节点分配颜色（物理寄存器）
4. **约束**：相邻节点不能有相同颜色

`build_interference_graph` 函数是寄存器分配算法的核心组成部分，通过构建冲突图为后续的图着色算法提供基础数据结构，是实现高效寄存器分配的关键步骤。
